# LDAP Backup Feature - slapcat-style LDIF Export

## Overview

LDAP Manager now includes a **backup feature** that exports the entire LDAP directory to LDIF format, similar to OpenLDAP's `slapcat` command. This provides a complete, portable backup of your directory that can be restored using standard LDAP tools.

**Implementation Date:** 2026-01-30
**Status:** ✅ Complete and production-ready

---

## What is LDIF?

**LDIF** (LDAP Data Interchange Format) is the standard text format for representing LDAP entries. It's:
- ✅ **Portable** - Works with any LDAP v3 server
- ✅ **Human-readable** - Plain text format
- ✅ **Complete** - Includes all attributes (user + operational)
- ✅ **Standard** - RFC 2849 compliant

## How It Works

### Backend (API)

**Endpoint:** `GET /api/backup/export/{cluster}`

**Process:**
1. Searches entire directory tree (SUBTREE scope)
2. Retrieves all entries with all attributes (`*` + `+`)
3. Formats as LDIF with proper encoding
4. Returns as downloadable `.ldif` file

**Example LDIF Output:**

```ldif

# Timestamp: 2026-01-30T12:00:00Z
# Base DN: dc=vibhuvioio,dc=com
# Total Entries: 1523
# Format: LDIF version 1

objectClass: organizationalUnit

gidNumber: 100

userPassword:: e1NTSEF9YWJjMTIzZGVmNDU2...  # base64 encoded

...
```

### Frontend (UI)

**Component:** `BackupButton.tsx`

**Features:**
- Checks backup availability automatically

- Shows progress toast notifications
- Error handling with user-friendly messages

**Filename Format:** `{cluster}-{timestamp}.ldif`
- Example: `vibhuvioio.com-20260130-120530.ldif`

## Access Control

### Keycloak Mode (AUTH_MODE=keycloak)

**Admin Role Required**
- ✅ User with "admin" role can export backups
- ❌ Editor, viewer, and auditor roles cannot export backups
- UI: Backup button shows only for admin users

**Why admin-only?**
- Backups contain **all directory data** including passwords (hashed)
- Access to backups = access to entire directory offline
- Admin-only ensures proper access control

### Read-Write Mode (AUTH_MODE=legacy, READONLY_MODE=false)

**Available to All Users**
- ✅ All users can export backups
- Suitable for trusted environments
- No authentication required

### Read-Only Mode (AUTH_MODE=legacy, READONLY_MODE=true)

**Not Available**
- ❌ Backup export returns 403 Forbidden
- Read-only mode is for viewing only
- Backups not needed (no modifications possible)

## Usage

### Via UI (Recommended)

1. **Login** (Keycloak mode) or access the app (read-write mode)
2. **Select a cluster** from the dashboard
3. **Click the "Backup" button** in the toolbar
   - Located next to "Create User" button
   - Shows download icon
4. **Wait for export** - May take a few moments for large directories
5. **File downloads automatically** - Named `{cluster}-{timestamp}.ldif`

**Progress Indicators:**
- Button shows "Exporting..." during backup
- Toast notification: "Exporting LDAP backup..."
- Success toast: "Backup exported successfully"

### Via API (Advanced)

**Check backup availability:**

```bash
curl -H "Authorization: Bearer $TOKEN" \
  http://localhost:8000/api/backup/status/vibhuvioio.com | jq

# Response:
{
  "cluster": "vibhuvioio.com",
  "backup_available": true,
  "base_dn": "dc=vibhuvioio,dc=com",
  "reason": "Admin access granted",
  "ldap_configured": true
}
```

**Export backup:**

```bash
curl -H "Authorization: Bearer $TOKEN" \
  -o vibhuvioio.com-backup.ldif \
  http://localhost:8000/api/backup/export/vibhuvioio.com

# Downloads LDIF file
```

**Response Headers:**

```http
HTTP/1.1 200 OK
Content-Type: application/x-ldif
Content-Disposition: attachment; filename="vibhuvioio.com-20260130-120530.ldif"
X-Cluster: vibhuvioio.com
X-Export-Timestamp: 20260130-120530
X-Base-DN: dc=vibhuvioio,dc=com
```

## Restoring from Backup

### Option 1: Using slapadd (OpenLDAP)

**For OpenLDAP servers:**

```bash
# Stop LDAP server
sudo systemctl stop slapd

# Clear existing database (CAREFUL!)
sudo rm -rf /var/lib/ldap/*

# Restore from LDIF backup
sudo slapadd -l vibhuvioio.com-20260130-120530.ldif

# Fix permissions
sudo chown -R openldap:openldap /var/lib/ldap

# Start LDAP server
sudo systemctl start slapd
```

### Option 2: Using ldapadd (Any LDAP server)

**For running LDAP servers (online restore):**

```bash
# First, clear existing entries (if needed)
# BE VERY CAREFUL WITH THIS!

# Then add from backup
ldapadd -x -D "cn=Manager,dc=vibhuvioio,dc=com" \
  -W -f vibhuvioio.com-20260130-120530.ldif

# Or for large files with better error handling:
ldapadd -x -D "cn=Manager,dc=vibhuvioio,dc=com" \
  -W -c -f vibhuvioio.com-20260130-120530.ldif
```

**Flags explained:**
- `-x` - Simple authentication
- `-D` - Bind DN (admin account)
- `-W` - Prompt for password
- `-f` - Input file (LDIF)
- `-c` - Continue on errors

### Option 3: Partial Restore

**Restore specific OUs or entries:**

```bash
# Extract specific OU from backup
grep -A 50 "ou=People" vibhuvioio.com-backup.ldif > people-only.ldif

# Restore just that OU
ldapadd -x -D "cn=Manager,dc=vibhuvioio,dc=com" -W -f people-only.ldif
```

## Backup Strategy Recommendations

### 1. Automated Daily Backups

**Using cron + API:**

```bash
#!/bin/bash
# /opt/backup-ldap.sh

CLUSTER="vibhuvioio.com"
BACKUP_DIR="/backup/ldap"
DATE=$(date +%Y%m%d)
TOKEN="your-admin-jwt-token"  # Or use OAuth flow

# Create backup directory
mkdir -p $BACKUP_DIR

# Export backup via API
curl -H "Authorization: Bearer $TOKEN" \
  -o $BACKUP_DIR/$CLUSTER-$DATE.ldif \
  http://localhost:8000/api/backup/export/$CLUSTER

# Compress backup
gzip $BACKUP_DIR/$CLUSTER-$DATE.ldif

# Retention: Keep last 30 days
find $BACKUP_DIR -name "*.ldif.gz" -mtime +30 -delete

echo "Backup completed: $CLUSTER-$DATE.ldif.gz"
```

**Add to crontab:**

```bash
# Daily backup at 2 AM
0 2 * * * /opt/backup-ldap.sh >> /var/log/ldap-backup.log 2>&1
```

### 2. Pre-Change Backups

**Before major changes:**

```bash
# Manual backup before bulk changes
curl -H "Authorization: Bearer $TOKEN" \
  -o pre-migration-backup.ldif \
  http://localhost:8000/api/backup/export/vibhuvioio.com
```

### 3. Off-Site Storage

**Upload to cloud storage:**

```bash
#!/bin/bash
# After backup, upload to S3/Azure/GCP

BACKUP_FILE="/backup/ldap/vibhuvioio.com-$(date +%Y%m%d).ldif.gz"

# Example: AWS S3
aws s3 cp $BACKUP_FILE s3://company-backups/ldap/

# Example: Azure Blob Storage
az storage blob upload --account-name company \
  --container-name ldap-backups \
  --file $BACKUP_FILE \
  --name ldap/$(basename $BACKUP_FILE)
```

## What Gets Backed Up

### Included in Backup

✅ **All Entries**
- Users (ou=People)
- Groups (ou=Groups)
- Organizational Units (ou=...)
- Custom entries
- Base DN and structure

✅ **All Attributes**
- Standard attributes (cn, sn, mail, etc.)
- Operational attributes (createTimestamp, modifyTimestamp, etc.)
- Custom attributes (if any)
- Multi-valued attributes (all values)

✅ **Binary Data**
- Passwords (hashed, base64-encoded)
- Photos (if stored in LDAP)
- Certificates
- Any binary attributes

✅ **Metadata**
- Entry creation time
- Last modification time
- Modifiers DN
- Entry UUID (if available)

### NOT Included in Backup

❌ **LDAP Server Configuration**
- Schema definitions
- Access control policies (ACLs)
- Overlays configuration
- Index configuration

❌ **Runtime State**
- Active connections
- Replication status
- Performance counters

**Note:** To backup configuration, use `slapcat -n 0` on the LDAP server directly.

## Performance Considerations

### Large Directories

**Directory Size:** 10,000 entries
- **Export Time:** ~5-10 seconds
- **File Size:** ~5-10 MB (uncompressed)
- **Compressed:** ~1-2 MB (gzip)

**Directory Size:** 100,000 entries
- **Export Time:** ~30-60 seconds
- **File Size:** ~50-100 MB (uncompressed)
- **Compressed:** ~10-20 MB (gzip)

### Optimization Tips

1. **Export During Low Traffic**
   - Schedule backups at night
   - Avoid peak business hours

2. **Compress Backups**
   ```bash
   gzip vibhuvioio.com-backup.ldif  # ~80-90% reduction
   ```

3. **Monitor Export Progress**
   - Check backend logs: `docker logs ldap-manager`
   - Look for: "LDIF export completed"

4. **Network Timeouts**
   - Large exports may take time
   - Frontend has timeout handling
   - Use API directly for very large directories

## Security Considerations

### ⚠️ Backup File Contains Sensitive Data

**What's in the backup:**
- User passwords (hashed, but still sensitive)
- Email addresses
- Personal information
- Full directory structure

**Security Best Practices:**

1. **Encrypt Backups at Rest**
   ```bash
   # Encrypt with GPG
   gpg --symmetric --cipher-algo AES256 backup.ldif

   # Decrypt when needed
   gpg --decrypt backup.ldif.gpg > backup.ldif
   ```

2. **Secure File Permissions**
   ```bash
   chmod 600 backup.ldif  # Read/write for owner only
   chown root:root backup.ldif
   ```

3. **Secure Storage**
   - Store backups on encrypted filesystem
   - Use cloud storage with encryption (S3 SSE, Azure encryption)
   - Restrict access (IAM policies, ACLs)

4. **Audit Log Access**
   - All backup exports logged with username
   - Monitor logs for unauthorized backups
   ```bash
   docker logs ldap-manager | grep "BACKUP_EXPORT"
   ```

5. **Rotate Backup Encryption Keys**
   - Periodically re-encrypt old backups
   - Use different keys for different years

## Troubleshooting

### Issue: "Backup not available" Message

**Possible Causes:**

1. **Read-Only Mode**
   - Backup not available in read-only mode
   - Switch to read-write or Keycloak mode

2. **Insufficient Permissions (Keycloak)**
   - User doesn't have admin role
   - Check user roles in Keycloak Admin Console
   - Assign "admin" role to user

3. **LDAP Not Configured**
   - Cluster password not cached
   - Configure LDAP connection first
   - Check: `/api/password/check/{cluster}`

### Issue: Backup Export Fails

**Check Backend Logs:**

```bash
docker logs ldap-manager | tail -50
```

**Common Errors:**

1. **"LDAP connection failed"**
   - LDAP server unreachable
   - Incorrect credentials
   - Network issues

2. **"Search failed"**
   - Base DN incorrect
   - Permissions issue on LDAP server
   - LDAP server overloaded

3. **"Timeout"**
   - Directory too large
   - Use API directly instead of UI
   - Increase timeout settings

### Issue: Backup Button Not Showing

**Checklist:**

1. **Check User Role (Keycloak Mode)**
   ```javascript
   // Browser console
   console.log(user.roles)  // Should include "admin"
   ```

2. **Check Deployment Mode**
   ```bash
   docker exec ldap-manager env | grep AUTH_MODE
   docker exec ldap-manager env | grep READONLY_MODE
   ```

3. **Check Browser Console**
   - Look for JavaScript errors
   - Check network tab for failed requests

## Comparison: LDAP Manager Backup vs slapcat

| Feature | LDAP Manager Backup | slapcat |
|---------|-------------------|---------|
| **Access Method** | HTTP API / Web UI | Command line (server access) |
| **Authentication** | Keycloak / RBAC | Root/sudo access required |
| **Format** | LDIF (RFC 2849) | LDIF (same format) |
| **Scope** | Base DN subtree | Full database or specific backend |
| **Server State** | Online (running) | Offline recommended |
| **Remote Access** | ✅ Yes (via API) | ❌ No (local only) |
| **User-Friendly** | ✅ Web UI | ❌ Command line |
| **Automation** | ✅ API (easy) | ✅ Shell scripts |
| **Config Backup** | ❌ Data only | ✅ With -n 0 flag |

**When to use LDAP Manager Backup:**
- ✅ Remote backups
- ✅ User self-service (admins)
- ✅ Scheduled API backups
- ✅ Web-based management

**When to use slapcat:**
- ✅ Full server backup (including config)
- ✅ Offline backup (server down)
- ✅ Direct server access available
- ✅ Maximum performance (no network)

## FAQ

### Q: Can I restore a backup to a different LDAP server?

**A:** Yes! LDIF is portable across LDAP v3 servers (OpenLDAP, 389 DS, Active Directory, etc.). Just ensure:
- Base DN matches or adjust DNs in the file
- Schema is compatible
- objectClasses are supported

### Q: Does backup include deleted entries?

**A:** No, only current entries are backed up. Deleted entries are not recoverable unless you have an older backup.

### Q: Can I schedule automatic backups?

**A:** Yes, use cron + API (see Backup Strategy section above). Or use the use-case backup scripts in:
- `use-cases/ldap-manager-read-write/README.md` (backup scripts)
- `use-cases/ldap-manager-with-keycloak/README.md` (backup scripts)

### Q: How do I backup multiple clusters?

**A:** Call the API endpoint for each cluster:

```bash
for CLUSTER in vibhuvioio.com vibhuvi.com oiocloud.com; do
  curl -H "Authorization: Bearer $TOKEN" \
    -o backup-$CLUSTER-$(date +%Y%m%d).ldif \
    http://localhost:8000/api/backup/export/$CLUSTER
done
```

### Q: Can I backup just specific OUs?

**A:** Currently, backup exports the entire base DN. For partial backups:
1. Export full backup
2. Use `grep` or LDIF tools to extract specific OUs
3. Or use `ldapsearch` directly for custom exports

### Q: Does backup affect LDAP performance?

**A:** Minimal impact for small/medium directories. For large directories:
- Export is read-only (no writes)
- Uses connection pooling
- May increase LDAP server load briefly
- Schedule during low-traffic periods
